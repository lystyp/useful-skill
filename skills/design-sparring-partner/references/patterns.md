# 設計模式對照表與「過度 / 不足」氣味清單

`design-sparring-partner` §4（抵抗過度工程）在判斷「該不該抽象、抽成什麼形狀」要深入時讀這份。

## 一、模式 ↔ 變化軸對照

設計模式不是裝飾，是「某條變化軸反覆出現時的具名解法」。先認出軸，再挑捕捉那條軸的模式。挑錯模式、或在沒有那條軸時硬塞模式，跟不抽象一樣是技術債。

| 變化軸（什麼東西會獨立改變） | 對應模式 | 介面長相 |
|---|---|---|
| 同一件事有多種**演算法 / 策略** | Strategy | `interface Policy { run(input): output }` |
| 接**外部系統 / 不可控邊界**（DB、API、檔案、Sheet） | Adapter / Port（Hexagonal） | `interface XxxPort { … }`，外部細節關在 adapter 裡 |
| **資料存取**來源會換（DB ↔ 記憶體 ↔ 遠端） | Repository | `interface XxxRepo { find/save/… }` |
| **建構**邏輯會變、或要依設定選實作 | Factory / 接線點 | `createXxx(config): Port` |
| 一份輸入要**通知多個下游** | Observer / Pub-Sub | `subscribe(handler)` |
| 一個流程的**步驟骨架固定、細節可換** | Template Method / Strategy 組合 | 骨架呼叫可替換的步驟 |
| 要在不改既有 code 下**加行為**（記 log、快取、權限） | Decorator / Middleware | 同介面包一層 |
| 多個物件要用**一致介面**操作、隱藏差異 | Facade | 對外一個窄介面，內部協調多個元件 |
| 物件**建立過程**複雜、參數多 | Builder | 分步組裝 |

挑模式的順序永遠是「軸 → 模式」，不是「我想用某模式 → 找地方塞」。

## 二、依賴方向（DIP）怎麼擺

- **高層**（orchestrator、use case、policy）定義它需要的介面，並只依賴介面。
- **低層**（adapter、具體儲存）去 implements 那個介面。
- 具體實作在一個**接線點 / wiring 檔**（`sources.ts`、`sinks.ts`、`container.ts`）被選定與組裝。換實作只動這個檔。
- 紅旗：orchestrator 直接 `import { SheetWriter }`。應該 `import type { ReportSink }`，具體在 wiring 注入。

## 三、過度耦合的氣味（該抽象卻沒抽象）

- 換一個外部系統 / 格式，要改的檔案散在好幾層。
- 高層模組 import 了某個具體 class / 某個 SDK 型別。
- 一個 function 同時知道「業務規則」跟「資料怎麼存的細節」（例如一邊算指標、一邊算 A1 cell 位置）。
- 測試高層邏輯時，被迫連真的 DB / 真的 API 一起拉起來。
- 使用者已經說「之後會接 X」，但 code 裡找不到那條 seam。

→ 解法：把會變的那條軸拉到介面後面，反轉依賴。

## 四、過度拆解 / 過度抽象的氣味（不該抽象卻抽了）

- 一個介面從頭到尾**只有一個實作**，而且沒有任何「未來會有別的」的訊號。
- 把「總是一起改」的東西拆成兩個模組，結果每次改功能都要同時動兩個檔（違反 common closure）。
- 一層 indirection 沒有捕捉任何變化軸，純粹「為了分層而分層 / 為了短而短」。
- 為了套某個設計模式，硬生出 `AbstractFactoryStrategyManager` 這種沒人需要的結構。
- 介面方法集過大（把所有實作各自的需要都塞進去），導致每個 adapter 都有一堆 `throw NotImplemented`。

→ 解法：砍掉投機抽象，改回具體但內聚的 code。需要時再抽，那時成本也不高。

## 五、判斷「一條軸是真是假」的速判

問三題，全否 → 假軸，不要抽象：

1. 現在就有 ≥ 2 個實作嗎？
2. 使用者明確點名了未來的實作嗎？
3. 這條軸的變動，是否會逼著我改動很多個地方（亦即不抽象的痛是真實且可預見的）？

只要任一為真 → 真軸，抽象它。三題都不確定時，**直接問使用者一句**「這個未來會不會有別的做法 / 別的來源 / 別的目標？」——比賭錯方向便宜太多。

## 六、SRP（單一職責）的操作型定義

不要用「一個 class 只做一件事」這種模糊講法。用 **「一個模組應該只為一個理由（一群會一起變的需求）而改變」**：

- 如果一個模組會因為「業務規則變」**和**「儲存格式變」兩個不同理由各自被改 → 該拆。
- 如果兩個模組總是因為同一個理由被一起改 → 該合。

職責的邊界 = 變化理由的邊界。這跟「變化軸」是同一件事的一體兩面。
