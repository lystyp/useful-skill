# Review Finding Format

一個 review 問題要讓**非作者**也能秒懂，光說「這裡有 race condition」不夠——他不知道什麼情況才會踩到、後果多嚴重、值不值得現在修。

本規範定義一個問題該怎麼寫。與工具、語言、平台無關；`code-review`（輸出問題清單）與 `pr-inline-comment`（發成留言）共用同一套，所以 review 完的內容可以直接拿去發。

## 四個必要成分

| 成分 | 作用 | 沒有它會怎樣 |
|------|------|-------------|
| **位置** | 檔案 + 行號範圍，可點擊 | 讀者要自己找，找錯地方就白吵一輪 |
| **問題** | 違反什麼 / 風險是什麼 | 只有情境沒有判準，變成講故事 |
| **🗣️ 白話情境** | 演一遍怎麼踩到 | 讀者無法判斷嚴重度，只能相信或不信 |
| **建議解法** | 一段最小 code snippet | 讀者知道有問題但不知道怎麼修 |

## 完整格式（獨立的問題清單）

````markdown
#### 問題 N：<一句話標題>

- **位置**：[檔名](相對路徑#L起-L迄)
- **類型**：<正確性 / 可讀性 / 可維護性 / 架構 / 安全性 …>
- **問題**：<說明；含多個風險時用編號列表拆開，不要擠成一段>
- **🗣️ 白話情境**：<場景一句開頭——>
  1. <事件步驟>
  2. <事件步驟>

  <**預期**：… **實際**：… 一行對比收尾>
- **建議解法**：

```
// 修正後
<code snippet>
```
````

## 精簡格式（已錨定在程式碼行上的 inline 留言）

留言本身就掛在那一行，不需要再重述檔案與行號脈絡。依序寫**三段**即可：

1. **問題** — 是什麼問題 / 風險（必要時帶觸發條件、為什麼）
2. **🗣️ 白話情境** — 見下節
3. **解法** — 建議怎麼修，附最小 snippet，**放在最後**

## 🗣️ 白話情境的寫法（不可擠成一行）

寫法依循 bug report「重現步驟」的業界慣例——編號步驟、一行一動作、最短路徑、預期 vs 實際收尾：

1. **開頭一句設定場景**（誰、在做什麼），以「——」收尾接列表
2. **事件用編號列表逐步展開**，一行一動作，只留重現問題的最短路徑；**順序由列表本身表達，不標 wall-clock 時間**。只有「時間長度本身是重點」時才標時間（部署空窗期多久、快取 TTL、timeout 視窗）
3. **並發 / 競態情境**用「工人 A / 工人 B」（或請求 A / 請求 B）的**交錯步驟**表達相對順序（例：A 讀 → B 讀 → A 寫 → B 寫蓋掉 A），不用絕對時間
4. **對比型情境**（同一個人做 A、B 兩個動作結果矛盾）改用兩條列表項，各自帶「動作 → 系統回應 → 當事人反應」
5. **結尾用一行「預期：… 實際：…」對比收尾**，點出落差與誰會受害

### 範例（順序型）

> 🗣️ **白話情境**：管理員對一筆已公開的違規內容按「下架」——
>
> 1. 第一步：狀態欄位 `reviewStatus=REJECTED` 成功寫入資料庫
> 2. 還沒跑到第二步的重算，行程剛好中斷（部署 / crash）
> 3. 資料庫停在「人工已拒絕，但公開狀態仍是 PUBLISHED」
> 4. 之後沒有任何排程會修復，要等下次剛好有人觸發重算
>
> **預期**：按完下架，該筆內容從列表消失。**實際**：內容照樣公開展示，後台卻顯示「已下架」。

### 範例（並發型）

> 🗣️ **白話情境**：兩個請求同時對同一筆庫存扣款——
>
> 1. 請求 A 讀到 `stock = 1`
> 2. 請求 B 也讀到 `stock = 1`（A 還沒寫回）
> 3. 請求 A 寫入 `stock = 0`，回「下單成功」
> 4. 請求 B 用自己手上的舊值寫入 `stock = 0`，也回「下單成功」
>
> **預期**：只有一個人搶到最後一件。**實際**：兩個人都收到成功通知，庫存卻只扣了一次。

## 注意事項

- **不確定就標「待確認」**，不要直接判定違規。
- **含多個風險時用編號列表拆開**，不要擠成一段——讀者會只看到第一個。
- 若另有「需請 PM／設計確認」的延伸點（設計缺口、規格待對齊等，非純程式問題），放在解法**之後**、用 `---` 分隔並標 `📋`，與開發者要動手的部分區隔開。
- 白話情境的角色請用**具體的人與動作**（「管理員按下下架」「使用者連點兩次送出」），不要寫成「當某個條件成立時」——後者等於沒演。
