# 註解品質 Code Review 規範

## 核心原則

本規範依循 Stack Overflow Engineering《Best practices for writing code comments》九條準則：
<https://stackoverflow.blog/2021/12/23/best-practices-for-writing-code-comments/>

核心立場：**註解只解釋「為什麼」（意圖、取捨、非顯而易見的限制），不複述「做什麼」**。編譯器不會檢查註解，過於繁瑣或複述程式碼的註解最容易在程式演進後失準，反而**誤導後續維護者與 AI**。寧缺勿濫——能用命名與結構表達的，就不要寫成註解。

> 若專案習慣以「意圖／取捨」型註解承載業務脈絡，這類**為什麼**註解應予保留；本規範針對的是「複述程式碼」「已過時」「過度繁瑣」的註解，而非要求移除所有註解。

九條準則摘要：

1. 註解不要複述程式碼（冗餘、徒增雜訊、且易過時）
2. 好的註解不該為「看不懂的程式碼」開脫（先改程式，而非加註解）
3. 寫不出清楚的註解，可能是程式碼本身有問題（考慮重構）
4. 註解應消除困惑、而非製造困惑（含已過時／與程式矛盾的註解）
5. 非慣用寫法（unidiomatic）要用註解說明原因
6. 複製來的程式碼要附原始出處連結
7. 在最有幫助處附上外部規格／文件連結
8. 修 bug 時補上「為何這樣修」的註解
9. 用標準標記（`TODO` / `FIXME`）標出未完成的實作

## 檢查清單

> 逐條使用 TodoWrite 工具標記檢查進度。

- [ ] **複述型註解**：註解只是把同一行程式碼用自然語言重講一遍（如 `i++ // 把 i 加一`，或 `// 設為 PRIVATE` 緊接 `visibility: PRIVATE`）→ 刪除（準則 1）
- [ ] **過時／矛盾註解**：註解描述與當前程式碼行為不符——提到已改名的變數／函式、已移除的旗標、已不存在的步驟、寫死的行號或數值 → 更新或刪除（準則 4；**這是誤導 AI 與維護者的最大來源**）
- [ ] **繁瑣的逐步流程註解**：用一長串註解逐行描述方法「怎麼做」，而程式碼本身已清楚表達 → 改以意圖命名的 private method 取代（交叉參照 `readability.md` 的 Composed Method；準則 3）
- [ ] **以註解掩蓋壞程式**：靠註解解釋一段晦澀程式碼在做什麼 → 先用清楚命名／拆解改善程式，而非加註解（準則 2、3）
- [ ] **被註解掉的死碼（commented-out code）**：版本控管已保留歷史，註解掉的程式碼一律刪除（準則 1）
- [ ] **缺少必要的「為什麼」**：非慣用寫法、繞過框架、刻意的 workaround、fail-safe／fail-closed 抉擇等缺少意圖說明 → 補上（準則 5、8）
- [ ] **未標記的未完成項**：暫時實作／已知缺口散落在一般註解或完全沒標 → 改用標準 `TODO` / `FIXME` 標記（準則 9）
- [ ] **引用缺連結**：引用外部規格／演算法／複製來源卻無連結 → 補上權威出處連結（準則 6、7）

## 建議格式

```
- **[註解]** {檔案}:{行數範圍}
  - 問題：{複述程式碼 / 已過時 / 過度繁瑣 / 死碼 / 缺少 why}
  - 準則：{對應九條準則編號}
  - 建議：{刪除 / 更新為意圖說明 / 抽成具名方法 / 補連結或 TODO}
```
