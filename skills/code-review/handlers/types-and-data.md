# 型別安全與資料形狀 Types & Data Smells Code Review 規範

> 適用所有原始碼檔，聚焦「型別系統有沒有被繞過」與「資料被建模成什麼形狀」。無靜態型別的語言，把「型別檢查」對應到 runtime 驗證與資料形狀的一致性。
> magic number 的「同值散落多處」重複面向見 `dry.md`（本檔管「單一字面值該不該具名」）；primitive obsession / data clumps 的「造成跨模組耦合」面向見 `coupling-cohesion.md`（本檔管其資料建模面向），互引避免重複。

## 核心原則

1. **不要用斷言繞過型別系統 —《Clean Code》/ 型別即契約**
   萬用型別、強制轉型、強制解包都是對檢查器說「相信我」，一旦判斷錯就是執行期 crash。信任邊界的資料應在入口以 runtime 驗證器檢查後，才取得可信的型別 / 形狀，而非硬斷言。

2. **具名優於字面值 —《Clean Code》Meaningful Names**
   有業務含義的數值 / 狀態字串該有名字（列舉 / 具名常數 / 設定檔），兼顧可搜尋性；狀態值一律走列舉而非散落字面值，業務常數集中於設定來源，共用型別 / 資料結構定義集中於專案既有的型別定義位置。

3. **用型別表達領域概念 — Primitive Obsession / Data Clumps（Fowler）**
   裸字串 / 數值表達 money / userId / 狀態、或總是成群出現的參數（頁碼 + 每頁筆數），宜升為具名型別——但第三次重複才做（Rule of Three）。

## 偵測訊號（code smells）

> 逐條使用 TodoWrite 工具標記檢查進度。

**型別安全（繞過）**
- [ ] **萬用 / 動態型別濫用**：函式邊界、資料存取回傳、外部 payload 標成「任意型別」或完全未標註而退化成動態——收斂成具名型別，或標成「未知型別」後顯式收窄
- [ ] **強制轉型繞過檢查**：把值硬轉成宣稱的型別（含連續兩次轉型繞過檢查器），尤其套在請求內容 / 反序列化結果 / 資料存取回傳上——改為 runtime 驗證通過後再取得型別
- [ ] **非空斷言 / 強制解包**：對「查單筆」「find 找元素」的結果直接假設非空就取用——改為顯式空值檢查 / guard（trust-boundary 護欄，非 YAGNI 可省）

**具名 / 常數**
- [ ] **狀態字串字面值取代列舉**：比對、賦值時直接寫 `'PENDING_SELECTION'` / `'ACTIVE'` 等裸字面值——改用列舉 / 具名常數集合
- [ ] **Magic number / string**：門檻、費用、重試上限、逾時毫秒、每頁筆數硬編在邏輯裡——移到設定來源或具名常數（同值散多處的重複面向見 `dry.md`）

**資料形狀**
- [ ] **Primitive Obsession**：裸字串 / 數值表達有領域含義的值（money / userId / token / 狀態），靠註解或前綴提醒單位——考慮 value object / 具名包裝型別（Rule of Three 才做）
- [ ] **Data Clumps**：起訖時間、頁碼 + 每頁筆數、經緯度反覆成對穿過多層簽章——收成具名型別（`DateRange` / `Pagination`）
- [ ] **Boolean blindness**：多個布林旗標並列於同一物件 / 簽章（`isActive` + `isMuted` + `isDeleted`）使呼叫端難辨語意——視情況改列舉狀態或拆流程
- [ ] **可 immutable 卻放任 mutation**：DTO / 設定物件 / 傳入的集合被就地修改；語言提供不可變宣告（唯讀欄位、凍結物件、不可變集合型別）卻沒用

## 重構手法對照

| 訊號 | 解法 |
|------|------|
| 萬用 / 隱性動態型別 | 具名型別，或「未知型別」+ 型別守衛收窄 |
| 強制轉型 | 以 runtime 驗證器驗過後再取得型別，別硬轉 |
| 非空斷言 / 強制解包 | 顯式空值檢查 / early return / 安全取值運算子 |
| 狀態字面值 | 收斂為列舉 / 具名常數集合 |
| magic number | Extract Constant 至設定來源（業務常數）或具名常數 |
| Primitive Obsession | value object / 具名包裝型別（第三次重複才抽） |
| Data Clumps | Introduce Parameter Object，收成具名型別並放到既有型別定義位置 |
| Boolean blindness | 改列舉狀態，或拆成不同流程 |
| 放任 mutation | 標為唯讀 / 凍結，回傳新值而非就地改 |

## 實務拿捏

- **斷言不是一律禁**：確實比檢查器更懂型別、且來源已驗過時可用；但套在**未驗證的外部輸入**上就是紅旗——那正是 runtime 驗證該上的地方。
- **動態型別語言的對應物**：沒有靜態檢查器時，紅旗改看「跨信任邊界的資料有沒有經過 runtime 驗證」與「同一概念的資料形狀在各處是否一致」（欄位名 / 選填性 / 單位在不同函式間漂移，等同型別被繞過）。
- **Primitive Obsession / value object 別過早**：第一次用裸字串沒問題，第三次重複同樣的驗證 / 格式邏輯才抽（Rule of Three），避免過度工程。

## 建議格式

標籤：`[Types]`（型別安全）、`[Data]`（資料形狀 / 具名）。

```
- **[Types]** {檔案}:{行數範圍}
  - 問題：{萬用型別/強制轉型/強制解包/字面值/data clump…，引用訊號}
  - 風險：{執行期 crash / drift / 難維護}
  - 建議：{runtime 驗證後取型別 / 收斂列舉 / 抽具名型別}
```
