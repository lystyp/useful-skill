# API 健壯性 Code Review 規範

## 職責定義

本規範針對單支 API 的「端到端流程」做健壯性審查，涵蓋效能瓶頸、潛在 Bug、高併發保護三個面向。審查範圍橫跨「請求入口 → 請求處理 → 商業邏輯 → 資料存取 → 回應路徑」整條呼叫鏈，而非單一分層。這些是概念角色而非特定框架的層名；專案若無對應分層，映射到最接近的位置。

## 審查方式

1. 從請求入口出發，追蹤整條呼叫鏈直到資料存取與回應
2. 讀取每一層的實際程式碼，理解完整流程
3. 依下方三大檢查清單逐條審查
4. 發現問題時，標記具體檔案與行數

## 檢查清單

> 逐條使用 TodoWrite 工具標記檢查進度。

### 1. 效能（Performance）

#### 1.1 資料庫查詢
- [ ] 無 N+1 查詢（迴圈內逐筆查單筆，應改為單次批次查詢：一次帶入整組 id 條件）
- [ ] 無冗餘查詢（同一筆資料在同一 request 中被查詢多次，應傳遞結果而非重新查詢）
- [ ] 分頁 API 由資料庫層做筆數限制與位移，未一次撈全部資料再於記憶體切片
- [ ] 只取必要欄位與關聯，未載入整棵關聯樹或整張表
- [ ] 查詢過濾 / 排序用到的欄位在資料庫 schema 有對應索引
- [ ] 總數統計與列表查詢若條件相同，考慮是否可合併或置於同一交易避免兩者不一致

#### 1.2 非同步處理
- [ ] 彼此無相依的多個非同步操作並行執行，而非逐一等待
- [ ] 長時間外部呼叫（第三方 API、檔案處理）評估是否需要非同步化（queue / event）

#### 1.3 回應大小
> 分頁回應慣例（不回裸陣列、統一「資料 + 分頁中繼資訊」結構）與「Response 不含多餘欄位 / 不外洩內部細節」見 `review-lessons.md` §7、`security.md`（Data-Exposure）。本檔不重複，避免規則 drift。

### 2. 潛在 Bug（Correctness）

#### 2.1 Null / 空值安全
- [ ] 「查單筆」的結果有查無資料的檢查，未直接存取屬性
- [ ] 可選欄位在使用前有存在性檢查
- [ ] 解構 / 展開取值時，來源物件不可能為空

#### 2.2 交易一致性
> 交易 context（連線 / session / handle）在各層之間的傳遞慣例見 `review-lessons.md` §6，本檔不重複。
- [ ] 多筆寫入操作（新增 + 更新、跨表寫入）包在同一個資料庫交易中
- [ ] 寫入後的讀取若需一致性，在同一交易內執行

#### 2.3 邊界條件
- [ ] 空集合輸入不會導致錯誤或非預期全撈（如「id 屬於空清單」條件的實際語意是否符合預期）
- [ ] 數值欄位有合理的邊界驗證（負數、零、超大值）
- [ ] 字串欄位考慮空字串 vs 空值的語意差異
- [ ] 分頁參數為 0 或負值時的行為是否正確

#### 2.4 錯誤處理
- [ ] 例外捕捉區塊未吞掉錯誤（捕捉後有重拋或 log）
- [ ] 業務例外使用正確的 HTTP status code（404 vs 400 vs 409 vs 422）
> 「錯誤訊息 / 回應不外洩內部細節」見 `review-lessons.md` §7，本檔不重複。

#### 2.5 競態條件
- [ ] 「先查再寫」模式（check-then-act）是否存在併發風險（如：檢查不存在 → 建立，兩個 request 同時通過檢查）
- [ ] 若有競態風險，是否用唯一性約束 + 攔截重複錯誤，或交易隔離等級保護
- [ ] 狀態機轉換（如 PENDING → APPROVED）是否把「預期的來源狀態」寫進更新的過濾條件做樂觀鎖
- [ ] 「讀-算-寫」重算衍生欄位（由其他欄位推導的狀態值）是否與「觸發重算的寫入」包在同一交易：分屬不同交易時，並發請求各自讀到過時快照、互相覆寫（正解：觸發寫入先取得 row lock、重算在同一交易內讀寫以序列化並發；其間的外部呼叫須留在交易外）

#### 2.6 授權 / 物件層級存取控制（BOLA）
> OWASP API Security Top 10 第 1 名（Broken Object Level Authorization）。屬 trust-boundary 護欄，不在 YAGNI 精簡範圍。完整訊號與修法見 `security.md`，本檔只做端到端追鏈時的最低限度確認。
- [ ] 帶資源 id 的端點在讀 / 寫前，驗證呼叫者對「該物件」有權（擁有者 / 已授權），而非只驗 token 有效——否則換個 id 即可存取他人資源（IDOR）
- [ ] 巢狀資源逐層驗歸屬：子資源確屬該父資源、父資源確屬該使用者
- [ ] 授權盡量以查詢條件落實（過濾條件綁 owner，查不到即視為無權），而非先撈全量再於應用層比對（兼省多餘查詢）
- [ ] 寫入端點未因信任前端而略過授權，與讀取端點一視同仁

### 3. 高併發保護（Resilience）

#### 3.1 資料庫保護
- [ ] 無 unbounded query（列表查詢缺筆數上限，可能在資料量大時拖垮 DB）
- [ ] 寫入密集的 API 評估是否需要 rate limiting（如檔案上傳、批次建立）
- [ ] 大量資料的批次操作使用分批處理（chunk），而非單次交易處理全部

#### 3.2 外部依賴保護
- [ ] 外部 API 呼叫設有 timeout / 取消機制
- [ ] 外部 API 失敗時有降級策略（回傳預設值、快取、或明確錯誤），而非讓 request hang 住
- [ ] 檔案上傳有大小限制（於上傳處理層或輸入驗證處設限）

#### 3.3 冪等性
- [ ] 建立資源的 API，若有自然鍵（如 name + type），使用唯一性約束防止重複建立
- [ ] 重試安全：同一 request 重送不會產生副作用（如重複扣款、重複發送通知）

## 發現問題後的處理

發現問題時，與使用者**逐一確認**每個問題的修改方向（是否要修、怎麼修、有無理由保留現狀），再進行修復——不要看到問題就直接動手改。
