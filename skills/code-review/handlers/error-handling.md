# 錯誤處理 Error Handling Code Review 規範

> 橫切視角：「錯誤這件事本身怎麼被**產生、傳遞、攔截、清理**」。適用所有語言、所有程式碼層。
> 文中「請求入口 / 商業邏輯 / 資料存取 / 共用工具」是**概念角色**，不是特定框架的層名；專案若無對應分層，映射到最接近的位置。
> 邊界（避免與既有 handler drift）：
> - `review-lessons.md` 管 **fail-loud 原則**（§2）與 **錯誤訊息脫敏 / 三處一致**（§7）；本檔從非同步 / catch / 錯誤型別角度補具體訊號、只互引不重列判準。
> - `api-robustness.md` 是「單一 endpoint 端到端追鏈」視角，管該端點 catch 有沒有吞、狀態碼對不對、以及**交易的併發 / 一致性設計**；本檔管**跨檔一致的錯誤處理模式**與**錯誤路徑上的 rollback / 資源釋放**。retry / timeout / idempotency 的韌性深水區歸 api-robustness，本檔僅點到（失敗要被接住、不能吞）。

## 核心原則

1. **用語言原生的錯誤型別承載錯誤，不要拋字串或裸資料結構 —《Clean Code》ch.7**
   錯誤型別才帶 stack trace、型別判斷契約與自訂欄位（如錯誤碼、對應的回應狀態）。用**屬性**區分情境，別為每種情境開一個子類。

2. **區分 operational 與 programmer error。** 可預期的（外部逾時、驗證失敗、查無資源）記錄 + 回語意化錯誤即可續行；非預期 bug（空值解參、invariant 被破壞）狀態已不可信，fail loud，別就地吞掉硬撐。

3. **Fail fast。** 壞輸入 / 被破壞的不變條件在入口立刻拋出（信任邊界用專案既有的驗證機制），愈早拋出愈接近病灶；別讓 buggy 輸入往下擴散。

4. **集中式錯誤處理：低層拋出、邊界統一轉協定層錯誤。** 資料存取與商業邏輯只拋專案既有錯誤型別、不碰框架的請求/回應物件、不決定回應狀態碼；請求入口不吞、讓錯誤冒泡到統一錯誤處理器單點對映。

## 偵測訊號（code smells）

> 逐條使用 TodoWrite 工具標記檢查進度。

**錯誤的型別與語意**
- [ ] 拋出的是字串 / 裸資料結構（`throw {code,msg}`），而非專案既有錯誤型別體系中的型別
- [ ] 統一拋通用未分類錯誤——可預期失敗應用有語意的錯誤型別，讓統一錯誤處理對映正確狀態碼
- [ ] catch 到底層錯誤（資料庫驅動 / HTTP client 的原生錯誤）後未包裝成專案錯誤型別就讓它原樣冒泡穿層

**吞錯 / 降級（swallow）**
- [ ] 空 catch（捕捉後無 rethrow、無處理）
- [ ] catch 後只記 log 就繼續往下跑，把失敗當成功（log 不算處理）
- [ ] catch 把錯誤降級成空值 / 空集合 / 預設值，害上游踩空或誤判「查無資料」
- [ ] catch 範圍過寬、catch-all 後憑猜處理，未依錯誤型別 / 錯誤碼分類
- [ ] 重拋時丟棄原因鏈，未保留原始錯誤（cause）與 stack

**非同步 / 併發**
- [ ] 未接住的非同步結果：回傳 future/promise/task 的呼叫未被等待、回傳或掛上失敗處理（尤其 fire-and-forget 的通知、log、訊息佇列送出）
- [ ] 等待動作不在 try 範圍內、或直接回傳未等待的非同步值，使失敗落在 try/catch 之外
- [ ] 請求入口的非同步失敗未進入統一錯誤處理（部分框架需顯式包裝 async handler 才會接住）
- [ ] 並行執行多個有副作用的子任務時，未評估「一個失敗、其餘已寫入」的部分成功問題（該逐一檢查結果 / 補償 / 包在同一交易）
- [ ] 迴圈內逐筆等待的失敗邊界不明：一筆失敗要中止整批還是續跑？例外從迴圈中央炸出、前面副作用無人清理

**資源清理 / 交易錯誤路徑**
- [ ] 開啟的資源（檔案 handle、外部連線、計時器 / 排程）在錯誤路徑上未釋放（缺 finally / defer / with 等語言對應機制）
- [ ] 交易區塊內的 catch 把該冒泡的錯吞掉，導致交易誤 commit（此處反而要確認 catch **沒有**吞錯）

**分層邊界**
- [ ] 資料存取 / 商業邏輯設定回應狀態碼或碰框架的回應物件（狀態碼只該在統一錯誤處理決定）
- [ ] 請求入口 try/catch 後自組錯誤回應繞過統一錯誤處理（無明確理由）
- [ ] 新增的錯誤型別未被統一錯誤處理對映（會 fall through 成 500）；對映的狀態碼與 API 文件 / 規格描述不一致

## 重構手法對照

| 訊號 | 解法 |
|------|------|
| 拋字串 / 裸資料結構 | 換成專案錯誤型別體系中對應的型別 |
| 未分類的通用錯誤 | 用語意化錯誤型別，讓統一錯誤處理對映正確狀態碼 |
| 底層原生錯誤穿層 | 在資料存取 / 商業邏輯邊界 catch + 型別判斷後包成專案錯誤型別 |
| 空 catch / log-only 續跑 | rethrow（或包裝後 rethrow）；真可忽略要在 catch 內註解「為何安全忽略」 |
| catch 降級成空值 | 讓錯誤冒泡；「查無」用正常回傳、「失敗」用拋出，語意分離 |
| 未接住的非同步結果 | 補等待（放進 try）或掛上失敗處理；fire-and-forget 顯式接住失敗 |
| 直接回傳未等待的非同步值 | 先等待再回傳，把失敗拉回 try 邊界 |
| 入口非同步失敗漏接 | 依框架慣例包裝 handler，把失敗導向統一錯誤處理 |
| 並行子任務部分成功遺失 | 改逐一檢查每個結果，或有副作用者包同一交易全成/全敗 |
| 錯誤路徑資源未釋放 | 用語言的 finally / defer / 作用域資源機制釋放；DB 用交易區塊讓拋出自動 rollback |
| 低層決定回應狀態碼 | 移除狀態碼，只拋錯誤型別，狀態碼收斂到統一錯誤處理 |

## 建議格式

```
- **[錯誤處理]** {檔案}:{行數範圍}
  - 情境：{什麼輸入 / 失敗路徑會觸發}
  - 問題：{吞錯 / 型別錯 / 非同步邊界 / 資源未釋放 / 分層錯，引用訊號}
  - 建議：{具體修法}
  - 連動：{新錯誤型別要在統一錯誤處理註冊 / API 文件狀態碼對齊等}
```
