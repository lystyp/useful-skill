# 跨模組設計／正確性 Code Review 規範（Review Lessons）

## 職責定義

本規範收錄從歷次人工 review 累積、**不屬於單一主題 checklist** 的設計與正確性重點。其他 handler 負責結構、職責與依賴；本檔負責「邏輯設計上容易出事、reviewer 反覆抓到」的跨模組問題。對任何 PR 都適用，與 `api-robustness.md` 互補（後者偏效能/併發/Bug，本檔偏狀態設計、可維護性、資料生命週期）。

下文的「請求入口／商業邏輯／資料存取／共用工具」是**概念角色**，不是特定框架的層名；專案若無對應分層，映射到承擔該職責的最接近位置即可。

## 檢查清單

> 逐條使用 TodoWrite 工具標記檢查進度。

### 1. 狀態機／狀態欄位一致性
- [ ] 多個狀態欄位（如 `status` / `reviewStatus` / `manualReviewStatus`）的語意是否解耦清楚，各自只表達一個 concern
- [ ] 操作（approve / reject / takedown / restore…）是否可能留下**矛盾組合**（如 `manualReviewStatus=APPROVED` 但 `status=UNPUBLISH`）
- [ ] 正向操作是否為反向操作的真逆操作（approve 應能完整還原 reject 造成的所有副作用，否則需要補 restore 入口或重新設計）
- [ ] 是否存在「進得去、出不來」的死狀態（如某狀態沒被任何白名單收，導致使用者永遠無法回復，只能改 DB）
- [ ] 能用「單一欄位升格為最高決策層 + 獨立 if 守門」取代多欄位交叉判斷時，優先簡化（減少狀態組合爆炸）

### 2. 函式名實相符 / 壞輸入 fail loud
- [ ] 函式行為與名稱一致：`toXxx` / `parseXxx` 等轉換函式不應有「原樣回傳」或「悄悄 fallback」的分支
- [ ] 壞輸入應 `throw` 明確錯誤（fail loud），不要默默產出壞結果（如把完整 URL 再 prepend 一次 domain 變成 404 連結）
- [ ] 「該不該做轉換」的判斷與「怎麼轉換」的實作職責分離：transformer 只負責轉換，要不要轉由 caller 決定
- [ ] 用旗標參數（如 `force=true`）蓋過原本行為時，先問是否該把職責拆乾淨，而非加參數繞過

### 3. 過渡相容用一次性 migration，不留雙軌邏輯
- [ ] 欄位搬家／資料模型調整時，優先寫**一次性 migration** 把舊資料搬到新位置，而非在程式裡長期保留 derive / 上下相容的 fallback 邏輯
- [ ] migration 後把對應的 derive util / 相容分支一併刪除（確認前端也無對應 UI 才算過渡完成）
- [ ] 過渡期應為「一次跑完」，不是「永久共存兩套」

### 4. 資料生命週期 / 外鍵刪除策略
- [ ] audit log、合規紀錄、金流/帳務紀錄**不可** `ON DELETE CASCADE`（違反不可變本質；user 一刪歷史證據全消失）。改用 `RESTRICT` / `SET NULL` 或軟刪除
- [ ] 設 CASCADE 前先問：父資料被刪時，這張表的資料是否真的該一起消失
- [ ] `staging` / 生成結果等「後台想回查」的中間資料，finalize 後是否需要保留而非清除

### 5. Refactor 後清死碼
- [ ] 重構/刪 method 後，原本的 import、模組、class、錯誤型別、型別定義、輸入驗證規則是否還有人用（全 repo grep 確認）
- [ ] 沒有任何 caller 的就直接刪除，不留無用死碼（reviewer 一定會抓）
- [ ] 「為了之後可能會用」而留的未使用 API / 參數，傾向先砍，真的有需求再開

### 6. 不為不存在的 caller 設計 API / 不破壞 invariant
- [ ] 參數化（如把寫死的 `source: MANUAL` 改成傳入參數）前，先確認是否會破壞既有業務 invariant（如「MANUAL 永遠保留、不被降級」）
- [ ] 不為「目前不存在的 caller」預先設計泛用介面（YAGNI）；參數只影響 create 卻不影響 update 分支會造成語意混亂
- [ ] 安全網式的固定查詢條件（如刪除時寫死只匹配 `source = MANUAL`，防誤刪 GENERATED 資料）不要為了泛用而拿掉
- [ ] 資料存取函式的交易上下文（transaction handle）該必填就必填，避免 caller 漏傳導致 atomic 默默退化成 non-atomic

### 7. Schema / 回應與實際輸出對齊
- [ ] 回應 schema／API 文件與實際回傳的欄位一致；多帶或少帶都要修（即使該端點在 runtime 不強制驗證回應，文件也要對齊；特別注意用展開／繼承語法整包帶出的欄位）
- [ ] 回應**不外洩**內部細節：錯誤明細、內部 ID、audit 欄位、完整關聯樹不回給前端
- [ ] schema 標必填的關聯欄位，對應的查詢要真的把該關聯撈出來（否則回傳會缺欄位、與文件不符）
- [ ] 錯誤訊息文字在測試案例、API 文件、實作三處一致

### 8. 測試要驗到「聲稱」的行為
- [ ] 測試名稱聲稱的行為要被 assertion 真的驗到（如測 fallback 成 `ZH_TW`，expect 就要檢查 `ZH_TW`，不能只檢查有回應）
- [ ] 該檢查的欄位不可遺漏（如 locale fallback 測試要斷言最終 locale 值）
- [ ] 斷言要打在「行為的結果」上，而非只確認流程有跑完（只斷言「沒拋錯」「有回應」不算驗到）

### 9. 回覆 review / 修正必須實證，不可宣稱或猜測
> 慘痛教訓：曾回覆某 comment「已修（import 分組已移動）」但實際沒動，reviewer 追問「還是沒修~?」；又曾把已不存在的 `503` 回應在 API 文件裡「改描述」保留，但該端點實際不會回 503（外部服務失敗對應的錯誤型別是 **502**），等於 document 一個不存在的狀態。根因都是「沒對著實際程式碼確認」。

- [ ] 回報某 review comment「已修」**之前**，必須重新讀回該檔案 / grep 確認改動真的落地，不可只憑記憶或意圖宣稱；作者回覆內容要與實際 diff 一致
- [ ] 描述任何行為（API 文件、函式文件註解、行內註解）前，先查**實際 runtime 行為**再寫：例如某錯誤回應的 HTTP status 要回頭看對應錯誤型別真正帶的 status code（外部服務失敗類→502、非法狀態類→500…），不可沿用舊描述或猜
- [ ] 行為不存在了就**整段移除**，不要為了「保留個 entry」而改描述硬留（如已不會發生的 `503` 直接刪掉，而非改成別的語意）
- [ ] 功能狀態改變後（如某 provider 從「不支援」變「支援」），grep 全 repo 把所有過時引用一次清掉：API 文件描述、前端型別註解、樣板檔註解、PR 描述與測試計畫，散落多處都要對齊
- [ ] 每改一處就讀回驗證一次（edit → re-read / grep），多檔修正結束後再做一次整體 verification pass，確認每一項都真的落地

## 建議格式

沿用其他 handler 的問題清單格式，並附上 reviewer 慣用的完整推理鏈，讓作者一次看懂為什麼要改：

```
- **[設計/正確性]** {檔案}:{行數範圍}
  - 情境：{什麼操作序列／輸入會觸發}
  - 影響：{會造成什麼錯誤狀態／壞結果／維護負擔}
  - 建議：{具體修法}
  - 連動改動：{schema / migration / 其他檔案需要一起改的部分}
```
