# Architecture First

## 這個 skill 在治什麼病

LLM 寫 code 有三個會悄悄累積技術債的預設傾向：

1. **沿用既有結構**，而不是退一步問「抽象該切在哪個層級」——於是在錯的層級加東西。
2. **YAGNI 慣性**：只有被明確要求第二個實作時才抽象——於是當「未來有別的實作」這件事已經是已知時，仍然寫死。
3. **停在使用者命名的那個 seam**，不主動找其他 seam——使用者說「解耦 A 跟 B」，就只解 A/B，沒問「還有什麼會獨立改變」。

結果就是：要嘛過度耦合（該換實作時整片要改），要嘛為了「看起來乾淨」過度拆解（一堆只有一個實作的介面、無謂的 indirection）。

**根因只有一句：抽象是由「變化軸」（axes of change）驅動的，但動手前沒有主動盤點變化軸。**

這個 skill 強制一個順序：**先盤點變化軸與邊界 → 設計介面與依賴方向 → 跟使用者確認 → 才實作**。

---

## 核心原則

### 抽象由「變化軸」驅動，不由「看起來該分層」驅動

一個 seam（接縫 / 介面）只在以下任一成立時才值得存在：

- 這條軸**現在就有 ≥ 2 個真實實作**（CSV 與 Sheet、Postgres 與 MySQL…），或
- 使用者**明確點名了未來的實作**（「之後會接 DB」）——點名 = 確認這條軸會變。

兩者皆無 → 不要抽象。保持具體，但**內聚**（把會一起變的東西放一起）。這就是避免過度拆解。

反過來：只要上面任一成立，這條軸就**一定**要放到介面後面，orchestrator 只依賴介面。這就是避免過度耦合。

### 變化軸常常不只一條，而且藏在不同層級

最容易犯的錯：某一層已經有抽象了（例如「儲存格式」CSV/Sheet 已抽象），就以為「抽象已經有了」，而沒看到**真正在變的是另一層**（例如「輸出媒介」grid / DB，這是更高一層、grid 概念在 DB 根本不適用）。

動手前一定要問：**「這需求裡，哪些東西會各自獨立改變？分別在哪個層級？」** 把它們列出來。

### 聽出「變化軸訊號」

使用者的話裡常常已經埋了軸，要主動接住：

- 「未來可能換成 X / 之後也許用 Y / 先用 A 但會接 B」→ 點名了一條軸，**必抽象**。
- 「讓 X 自己決定怎麼做 Y」→ X 是一個封裝了策略的抽象（物件 / 介面），不是一個 free function。
- 「這部分可能有很多種做法」→ Strategy 軸。
- 「之後別人會接他們自己的 X」→ 對外的 port，必抽象。

漏接這些訊號，是最常見的失誤。

### 依賴方向：高層依賴抽象，不依賴細節（DIP）

orchestrator / policy / 高層流程**只能 import 介面**，不准 import 具體 adapter。具體實作在「接線點」（一個 wiring 檔，像 `sources.ts` / `sinks.ts`）被組裝起來，換實作只動那一個檔。

## 流程（硬閘門）

**在跟使用者確認架構之前，不要寫任何實作 code。** 這道閘門是這個 skill 的重點——使用者要的就是「反覆確認邊界與架構之後才實作」。理由：架構錯了，寫越多越難回頭；而盤點 + 確認的成本，比起改錯方向的成本，幾乎可以忽略。

### Phase 0 — 釐清功能與職責
- 這個功能「做什麼」與「不做什麼」，用一句話講清楚它的單一職責。
- 它的輸入、輸出、副作用（碰 DB？寫檔？呼叫外部？）。

### Phase 1 — 盤點變化軸
- 列出「哪些東西會各自獨立改變」。對每一條，標記它在哪個層級。
- 對每條軸判定：**現在就有 ≥2 實作？／使用者點名了未來實作？** 兩者皆無的軸，先不抽象（記下來，註明「暫不抽象，理由：YAGNI」）。
- 回頭掃一遍使用者的原話，有沒有漏接的軸訊號。

### Phase 2 — 決定邊界與介面
- 對每條「要抽象」的軸：seam 切在哪？介面（port）長什麼樣——**最小**方法集合是什麼？
- 標出依賴方向：誰依賴誰。確認高層只依賴介面。
- 選用對應的設計模式（見 `references/patterns.md`），並說明「為什麼是這個模式」。模式是「具名的邊界解法」，不是裝飾——選錯模式或硬塞模式，跟不抽象一樣糟。

### Phase 3 — 過度 / 不足自檢
逐項問（細節與更多 smell 見 `references/patterns.md`）：
- 每個介面都「賺到了它的存在」嗎（≥2 實作或點名的未來）？只有一個實作又沒被點名 → 砍掉，改回具體。
- 有沒有把「總是一起變」的東西拆開了（過度拆解 / 違反 common closure）？→ 合併。
- 有沒有具體實作洩漏進 orchestrator（過度耦合 / 違反 DIP）？→ 反轉。
- 每個模組的職責內聚嗎（SRP）？有沒有一個模組為了好幾個不同理由而改？

### Phase 4 — 把設計攤給使用者確認（閘門）
用下面的「設計提案範本」把架構講清楚，**明確等使用者點頭**。如果有沒想清楚的邊界，這裡用問題逐一釐清——一次問一個關鍵決定，不要一次倒一堆。

### Phase 5 — 實作
- 使用者確認後才動手。
- 一次接一個 adapter；先讓 orchestrator 依賴介面跑通，再補其他實作。
- 實作中若發現當初的邊界判斷有誤（例如冒出一條沒料到的軸），停下來、回 Phase 4 重新確認，不要默默改架構。

## 設計提案範本

跟使用者確認時，用這個結構（依需求大小伸縮，小需求可以濃縮成幾行）：

```
## 功能與職責
<一句話：這東西做什麼、不做什麼>

## 變化軸（axes of change）
- 軸 A：<會怎麼變> — 現在有 <幾個> 實作 / 使用者點名未來會 <…> → 要抽象
- 軸 B：<會怎麼變> — 目前只有一種、也沒被點名 → 暫不抽象（YAGNI）

## 介面設計
interface XxxPort {
  methodA(...): ...   // 為什麼需要這個方法
}
<每個介面的最小方法集，以及它捕捉的是哪條軸>

## 模組地圖與依賴方向
orchestrator ──depends on──▶ XxxPort（介面）
                                 ▲
                  SheetXxx ──implements──┘
（高層只依賴介面；具體在 wiring 檔組裝）

## 刻意不抽象的部分（以及為什麼）
- <某某> 保持具體，因為只有一種實作、也沒有未來變化的訊號。

## 用到的設計模式（以及為什麼）
- <Strategy / Adapter / Repository…>：對應 <哪條軸>

## 還沒決定 / 想跟你確認的
- <關鍵邊界問題，一次一個>
```

然後明確說：**「這個架構你 OK 嗎？確認後我才開始實作。」**

---

## 動手前自檢清單

寫實作 code 之前，這幾題要能答出來（答不出 = 還沒到 Phase 5）：

1. 變化軸有哪些？（列得出來）
2. 每條軸是真的嗎？（≥2 實作 或 被點名）假的軸有沒有被我誤抽象？
3. 每條真軸的 seam / 介面在哪？方法集是不是最小？
4. 依賴方向對嗎？（高層 → 抽象；orchestrator 沒有 import 具體 adapter）
5. 我**刻意不抽象**的是什麼？理由講得出來嗎？
6. 有沒有把「總是一起變」的東西拆開、或把「為不同理由變」的東西綁在一起？
7. **使用者確認架構了嗎？** ← 沒有就停在 Phase 4，不要往下寫。

---

## 平衡：不要矯枉過正

這個 skill 的閘門是為了「想清楚再動手」，不是為了「每件事都開會」或「每個東西都包介面」。

- 一兩行的瑣事、改個變數名、修 typo、補註解——**不要**套這套流程，直接做。
- 過度抽象（投機性介面、只有一個實作的 port、為了模式而模式）跟過度耦合**一樣是病**。Phase 3 的自檢就是用來砍掉投機抽象的。
- 當你不確定一條軸是真是假：**問使用者**「這個未來會不會有別的實作？」——一句話就能決定要不要抽象，比賭錯方向便宜太多。

深入的設計模式對照表與「過度/不足」氣味清單見 `references/patterns.md`。
