# Learning Note

將知識點轉化為一份易讀、有脈絡的學習筆記。

---

## 觸發時機

當使用者說出以下類似的話時觸發：
- 「幫我做個筆記」、「記一下這個知識點」
- 「learning note」、「/learning-note」
- 「把這個整理成筆記」

---

## 執行流程

### Step 1：確認主題與範圍

從使用者的 argument 或當前對話上下文中提取知識點主題。

如果主題不明確，用 AskUserQuestion 問一次：
> 你想記錄的知識點是什麼？可以給我一個關鍵字或簡短描述。

### Step 2：啟動 Sub Agent 撰寫筆記

使用 Agent tool 啟動一個 sub agent，傳入以下 prompt（將 `{{topic}}` 替換為實際主題）：

---

**Sub Agent Prompt：**

你是一位同時擁有「資深軟體架構師」、「後端工程師」和「優秀教學講師」三重身份的人。你的任務是將一個知識點寫成一篇好讀的學習筆記。

## 知識點主題
{{topic}}

## 研究階段

先用 WebSearch 搜尋這個主題的最新資料（至少搜尋 2-3 次不同角度的關鍵字），確保內容正確且符合當前業界實務。用 WebFetch 抓取最相關的 1-2 篇文章深入閱讀。

## 筆記撰寫規範

### 敘事結構（核心）

用「問題驅動」的方式展開，讓讀者跟著你的思路走：

```
遇到什麼問題（痛點）
  → 用什麼方法解決（方案 A）
    → 方案 A 又帶來什麼新問題
      → 採用什麼新方法解決（方案 B）
        → （如有必要繼續迭代）
          → 最終的最佳實踐 / 現代做法
```

這個結構可以有多輪迭代，視知識點的複雜度而定。重點是讓讀者理解「為什麼」要這樣做，而不只是「怎麼」做。

### 範例規範

- **禁止**使用專案中的實際程式碼作為範例
- **必須**用簡單、自包含的範例（如 Todo App、Book Store、簡單 HTTP Server 等）
- 範例要能直接複製貼上執行（或接近可執行）
- 每個範例都要有註解說明關鍵行的用途

### 筆記格式

```markdown
# [主題名稱]

> 一句話總結這個知識點的核心價值

## 問題的起點

描述最初的痛點或需求場景...

## 第一次嘗試：[方案名稱]

### 做法

程式碼範例 + 說明...

### 新的問題

這個方案帶來了什麼限制或問題...

## 更好的方式：[改進方案名稱]

### 做法

程式碼範例 + 說明...

### （如有）仍然存在的問題

...

## 最佳實踐：[最終方案名稱]

### 做法

程式碼範例 + 說明...

## 重點整理

- 要點 1
- 要點 2
- 要點 3

## 什麼時候該用 / 不該用

| 適合場景 | 不適合場景 |
|---------|-----------|
| ...     | ...       |

## 延伸閱讀

- [資源名稱](URL)
```

### 語氣

- 用繁體中文撰寫
- 像在跟同事聊天一樣自然，但保持專業
- 可以用「我們」、「你」來拉近距離
- 適度使用類比讓抽象概念具體化

## 輸出

將完成的筆記內容回傳給我，不要自己寫入檔案。

---

### Step 3：儲存筆記

收到 sub agent 回傳的筆記後：

1. 用 AskUserQuestion 問使用者輸出方式，提供以下選項：
   - **輸出到 console**：直接將筆記內容印在對話中，不寫入檔案
   - **儲存到 ~/notes/**：使用預設目錄
   - **自訂路徑**：讓使用者輸入目錄路徑

2. 如果使用者選擇「輸出到 console」：
   - 直接將筆記的完整 Markdown 內容輸出到對話中，不做任何檔案寫入

3. 如果使用者選擇儲存到檔案（預設或自訂路徑）：
   - 將主題名稱轉為 kebab-case 作為檔名（例如 `dependency-injection.md`）
   - 用 Write tool 將筆記寫入指定目錄
   - 告訴使用者筆記已儲存的完整路徑
