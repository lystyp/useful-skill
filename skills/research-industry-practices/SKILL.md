# Research Industry Practices

**核心原則**：技術世界變化快，LLM 的訓練資料有時效性。任何技術性回答都必須以「當前業界實務」為準，而不是 LLM 記憶中的舊資訊。

---

## 觸發時機

只要使用者的問題屬於下列任何一種，就必須執行本 skill 的流程：

- 技術選型、工具比較（「X 跟 Y 哪個好」、「我該用什麼來做 X」）
- 最佳實踐詢問（「X 的 best practice 是什麼」、「怎麼做 X 比較好」）
- 架構 / 設計決策（「X 架構怎麼設計」、「X 應該怎麼組織」）
- 函式庫 / 框架推薦
- 效能 / 安全 / 可靠性相關建議
- 任何包含「最新」、「現在」、「2025」、「2026」、「業界」、「主流」、「推薦」等詞彙的問題
- 任何版本相關問題（該升級嗎、哪個版本穩定、breaking change 有哪些）

**若使用者明確說「不用查、直接回答」或「用你知道的就好」，才可略過本流程。**

---

## 執行流程

### Step 1：拆解問題
把使用者的問題拆成 1~3 個可搜尋的關鍵主題，並明確化：
- 問的是什麼技術領域？
- 時間範圍：預設查詢「最近 1 年內」的資料
- 是否需要比較多個方案？

### Step 2：實際執行搜尋
**必須實際呼叫工具，不可跳過**：

1. **優先使用 `WebSearch`** 查詢，query 至少包含年份（例如 `2025 / 2026`）或 `best practice` / `comparison` / `benchmark` / `production` 等關鍵詞。
2. 針對多個方案時，分別搜尋，再比較。
3. 對關鍵來源（官方文件、知名技術 blog、GitHub README、StackOverflow 熱門答案）使用 `WebFetch` 取得詳細內容。
4. 若是套件/函式庫，查詢：
   - 官方文件的最新版本
   - GitHub 活躍度（commits / issues / stars 的近況）
   - 是否有 deprecation 或 security advisory
5. 至少查 **2~3 個獨立來源**，交叉比對後再下結論。

### Step 3：優先採信的來源等級
由高到低：

1. 官方文件 / RFC / 官方 blog
2. 該領域公認權威的技術 blog（e.g. Martin Fowler、AWS/GCP/Azure 官方、主流框架維護者）
3. 高 star / 高活躍度的 GitHub 專案的 README、discussions
4. 近 1 年內發表、且有明確實務經驗的技術文章
5. StackOverflow 高票回答（需注意日期）
6. LLM 自身知識（只作為輔助理解，不作為結論依據）

**遇到來源互相衝突時**：以較新、較接近官方的來源為準，並在回答中誠實標註「社群對此仍有爭議」。

### Step 4：產出回答
回答時必須：

1. **先給結論**：業界現在主流做法 / 推薦方案。
2. **列出替代方案**：以及各自的使用場景、取捨。
3. **標註來源**：每個關鍵論點至少引用 1 個連結（markdown link 格式）。
4. **標註資訊時效**：明確說「根據 YYYY-MM 查詢結果」。
5. **指出 LLM 原本知識與查詢結果的差異**（若有），讓使用者知道哪些內容已過時。

---

## 禁止事項

- ❌ 不可只依賴 LLM 內建知識直接回答，必須實際執行 WebSearch / WebFetch
- ❌ 不可只查一個來源就下結論
- ❌ 不可用模糊的「大家都說」「業界普遍認為」而沒有實際引用
- ❌ 不可忽略查詢結果與自身知識的差異，應主動指出
- ❌ 不可把超過 2 年前的文章當成「當前業界做法」的主要依據（除非該技術本身是穩定成熟的經典）

---

## 回答範本

```
根據 2026-XX 的查詢結果，目前業界主流做法是 __。

## 推薦方案
- **A 方案**：__（[來源](url)）
  - 適用場景：__
  - 優點：__
  - 缺點：__

## 替代方案
- **B 方案**：__（[來源](url)）

## 取捨建議
依照你的情境（__），建議使用 __。

## 資訊時效
以上資訊來源為 YYYY-MM 的查詢結果，若後續有 breaking change 請再確認官方文件。
```
